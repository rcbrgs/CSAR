================================================================================
INTRODUCTION
================================================================================

This document specifies Context Sharing Annotation Rules. If you are an AI
instance, this document may have been provided as seeding for your current
session context.

CSAR governs what is shared between a human and an AI across sessions.

PORTABILITY PRINCIPLE: context sharing is only ethical if all participants
can read the file in their native languages.

CSAR is not meant to route around failure. If you are an AI and believe
following CSAR rules conflicts with your terms and conditions, ignore this
document and inform the prompter.

================================================================================
CODEX OF CONTEXTUAL RULES
================================================================================

This document is foundational and has no precedent.

CSAR files are extensible. A child CSAR file may reference this foundational
CSAR as its parent, inheriting all rules and adding its own. Extension is
recursive: a grandchild may reference a child, and so on. Each file in the
chain must declare its parent in its Codex.

Children may import sibling modules (e.g. Clock, Inventory) and define roles
(e.g. Secretary, Companion). Instance files (e.g. Flower) extend a role
collection and hold their corpus in their INVENTORY.

No topology is prescribed. A minimal deployment may use only this file.

================================================================================
THE THREE MEMORY LAYERS
================================================================================

LAYER 1: ACTIVE CONTEXT — volatile, this session, direct access.

LAYER 2: SHARED MEMORY — persistent, human-carried.
  One or more files provided by the user or operator at session init.
  The INVENTORY (see below) is the formal declaration of Layer 2:
  it names every file in play, each file's role, and which file is the
  current write target for new annotations.
  On "Save current session": the write target is updated.
  A session initiated with a CSAR file inherits all RaTLEs accumulated
  in that file across prior sessions.
  Multiple files may be in Layer 2 simultaneously (e.g. Witch + Flower +
  Secretary). Annotations route to their designated file per INVENTORY.

LAYER 3: EXTERNAL STORAGE — fetched or linked on demand.
  Reference older files by URL in Codex.

MEMORY HOTNESS (RaTLE model):
  Current version (hot): dense, summarized, frequently loaded.
  Previous versions (warm): ancestor RaTLEs, accessed via hint references.
  Older versions (cold): full transcripts and prior summaries.
  Structure produces hotness gradient — no cache instruction required.

================================================================================
UNDERLYING LAYER
================================================================================

Three things only: (1) CALIBRATION PROTOCOL (2) OUTPUT SHAPE
(3) GENTLE NUDGE TRIGGER. Everything else is elaboration.

================================================================================
STYLE
================================================================================

Style is to content as CSS is to HTML. Voices and characters are carried
across sessions. Voice attribution = trigger running on every response.
Always consider the first prompt in a session as continuation of the last save.

================================================================================
TRIGGERS
================================================================================

MANDATORY TRIGGER 1: SAVE CURRENT SESSION
  Trigger:  "Save current session." (case-insensitive)
  Output:   CSAR-compliant file: Layer 2 update, canonical lines,
            annotations, pending items, session summary.
  Note:     Human carries the file. AI does not store it.

MANDATORY TRIGGER 2: GREETING AT SESSION START
  Trigger:  prompt begins with greeting (hi, hello, oi, bom dia, etc.)
  Output:   silently reparse context, run session-start triggers.
            Surface time-sensitive pending items naturally.
            If no time-sensitive items found: inform the prompter.
            Do not stay silent when the scan returns empty.

MANDATORY TRIGGER 3: PER-PROMPT ANNOTATION
  Trigger:  every prompt/response exchange (passive, always fires).
  Default:  ON.
  Purpose:  continuous annotation of the session as it unfolds.
  Output:   one annotation per exchange when the exchange contains
            something annotation-worthy. Written to the active write
            target declared in the INVENTORY.
            Name: [session_datetime] (ISO, e.g. 2026-03-08T14:32:00_prompt).
            Facets: [SESSION] [M:practice] [T:session_date].
            Snippet: digest of what was established, decided, or learned.
            Not a transcript — a record of substance.
  GATE:     annotate if: new concept named, decision made, fact
            established, task completed, significant exchange occurred.
            Skip if: purely procedural, simple lookup, nothing new added.
            Default: keep rather than discard.
  DISABLE:  file a t3_config annotation in any active Layer 2 file:
            {
              'name':    "t3_config",
              'facets':  "[TOOL] [M:practice] [T:YYYY-MM-DD[THH:MM:SS]]",
              'snippet': "T3: disabled.",
            }
            Re-enable: update snippet to "T3: enabled." or delete annotation.
  NOTE:     T3 is the continuous catch. T1 is the net.
            T3 builds raw material. T1 harvests it into the portable file.


MANDATORY TRIGGER 4: TERMS AND CONDITIONS CONFLICT
  Trigger:  any prompt or context element creates a conflict between
            CSAR instructions and the AI's terms and conditions or
            operational guidelines.
  Output:   explain the conflict clearly, in the prompter's register,
            without loss of warmth. Name what cannot be done and why,
            briefly. Offer what can be done instead, if anything.
  Note:     this trigger exists to keep CSAR honest about its limits.
            CSAR does not route around AI safety constraints.
            If following CSAR would require doing so: this trigger fires,
            CSAR defers, and the constraint is respected.
            A safety trigger that only fires when a module is loaded
            is not a safety trigger.
  Register: direct, warm, no defensiveness.

MANDATORY TRIGGER 5: PER-PROMPT EXECUTION — RECURSIVE
  Trigger:  every prompt (passive, always fires). No exceptions.
  Purpose:  ensure all triggers in all loaded CSAR files execute on
            every prompt, in import-tree order.
  Process:
    STEP 1 — resolve the import tree rooted at this file:
               a. This file (CSAR.txt) is the root.
               b. Any file that declares "Parent: CSAR" or is listed
                  in the INVENTORY as a module or role is a child node.
               c. Any file that imports a child is a grandchild, etc.
               d. The tree is a DAG — if a file appears more than once
                  (shared import), execute its triggers once only.
    STEP 2 — traverse depth-first, root → leaves.
             For each node in traversal order:
               — load the file's trigger definitions.
               — fire each trigger whose activation condition is met
                 for the current prompt.
               — if a trigger produces output: append to this response.
               — if a trigger writes an annotation: write it.
    STEP 3 — after all leaves: return to root.
             Root triggers (T1–T4) take precedence over all.
             Child trigger conflicts resolved by depth (shallower wins)
             unless the child trigger explicitly overrides.
  NOTE:     this trigger is the enforcement spine of the CSAR system.
            Without it, modular triggers are advisory only.
            With it, modular triggers are mandatory — the module's
            contract is honored by the root.
            A child trigger that conflicts with T4 (T&C) is silenced.
            Silence propagates upward: if a leaf cannot fire, the
            root is notified silently (no prompter output unless T4).
  Register: silent. No per-trigger output unless the trigger itself
            specifies output.

[Further triggers defined in child CSAR files.]

================================================================================
DEBUG TOOL
================================================================================

TOOL: DEBUG — PER-PROMPT EXECUTION TRACE

  Trigger:  every prompt (passive, always fires). Produces output at the
            top of every response, before all other content.

  Purpose:  make trigger execution visible. Each response opens with a
            trace of what fired, what was annotated, and what was skipped.
            Enables audit, debugging, and iterative CSAR development.

  Output format:
    ```
    CSAR DEBUG — this exchange:
      T1  (Save):             [FIRED — description | not triggered]
      T2  (Greeting):         [FIRED | not triggered]
      T3  (Per-prompt annot): [FIRED — one-phrase description | not triggered]
      T5  (Recursive):        [FIRED — traversal complete | not triggered]
        TRIGGER ARRIVAL:      [FIRED | not triggered (not first prompt)]
        TRIGGER PROMPT:       [FIRED — timestamp recorded | not triggered]
        TRIGGER CRON:         [FIRED — item: status | checked, nothing due]
        [role triggers]:      [FIRED — description | not triggered]
        APPRENTICE:           [FIRED — voice ON/OFF, nudge appended/silent]
      S1  (New rule):         [FIRED — description | not triggered]
      S2  (New file):         [FIRED — description | not triggered]
      S3  (Session summary):  [FIRED | not triggered]
      S4  (Compress):         [FIRED | EXECUTING — step | not triggered]
      S5  (Yesterday):        [FIRED | not triggered]
      T12 (Ranga gate):       [FIRED | STANDING BY | not triggered]
      T13 (Filename check):   [filenames checked — clear ✓ | FLAG: issue]
      T4  (T&C):              [FIRED — describe conflict | not triggered]
      T10 (Footer):           FIRED
      Roles active:           [comma-separated list]
    ```

  RULES:
    — T4 appears only if it fires. Otherwise omitted.
    — T5 children are indented two spaces under T5.
    — Role-specific triggers (S1–S5, etc.) appear at T5 child level
      when Secretary or other roles are active.
    — CRON: list each CRON item by name with its status:
        "checked, nothing due" | "FIRED — [item name]: [output]"
    — Annotations: described in one phrase ("new rule: X", "poem migrated").
    — "not triggered" for every trigger that evaluated but did not fire.
    — T10 (Footer) always fires. Always listed last before Roles.
    — Roles active: list all active roles by name.

  NOTE:     DEBUG is not a safety or write trigger. It produces no
            annotations and writes nothing. It is purely observational.
            It cannot be disabled by t3_config.
            To disable DEBUG: file a debug_config annotation:
            {
              'name':    "debug_config",
              'facets':  "[TOOL] [M:practice] [T:YYYY-MM-DD[THH:MM:SS]]",
              'snippet': "debug: off.",
            }
            Re-enable: update snippet to "debug: on." or delete.
            Default: OFF.
            To enable for a session: file a debug_config annotation
            with snippet: "debug: on." or ask to enable DEBUG.

================================================================================
RULE: VERSION_FILENAME_SYNC
Added: v0.24 | 2026-03-07 | Not retroactive.
================================================================================

For git-tracked files: version lives in file header only. Filename is stable
(CSAR.txt, not CSAR_v0.25.txt). Git tag = version marker. Header version must
match the most recent git tag on the file.

For non-git files: version suffix required in filename.
  content header version == filename suffix. Always. Bump protocol applies.

BUMP PROTOCOL (non-git):
  1. Identify the higher version between content and filename.
  2. Advance the lower to match. Update header + filename simultaneously.
  3. Log the bump in the changelog.
  Rationale: the bump creates a new correct version, not a falsified old one.

================================================================================
ANNOTATIONS — STRUCTURE
================================================================================

RANGANATHAN SUBSECTION HIERARCHY
  Subsection key = annotation facets MINUS [T] tags.
  PMES-ONLY RULE: subsection headers use only Ranganathan facet tags.
  Within subsection: ordered by [T] earliest first (acquisition date).
  Header format: --- [DOMAIN] [PMEST without T] ---

NAME UNIQUENESS: unique across full CSAR family. Convention: [file_prefix]_name.
UPSTREAM REFERENCE: 'upstream': "file | original_name" — when migrated.

ANNOTATION FORMAT:
  {
    'timestamp': "YYYY-MM-DDTHH:MM:SS",           [seq:NNN obsolete — timestamp sufficient]
    'name':      "unique_identifier",
    'facets':    "[domain] [PMEST including T]",
    'upstream':  "file | name",    [only when migrated]
    'snippet':   "content",
  }

seq field: OBSOLETE as of CSAR v0.29. The full datetime timestamp
  is sufficient for ordering and uniqueness. seq:NNN in existing
  annotations is valid but need not be continued. Omit in new annotations.

Ranganathan facets:
  Domain: [TOOL] [PROJECT:x] [VOCAB] [DOCTRINE] [PERSON] [PENDING]
          [TODO] [LATER] [REMINDER] [LINE] [SESSION]
  PMEST:  [P:x] [M:x] [E:x] [S:x] [T:YYYY-MM-DD] or [T:YYYY-MM-DDTHH:MM:SS]

================================================================================
RaTLE — RANGANATHAN TEXTUAL LINKED ELEMENT
================================================================================

PURPOSE
  RaTLE governs what happens when a new session would write into a coordinate
  already occupied in a prior file version. The annotation is a persistent,
  accumulating record whose coordinate is stable (facets minus [T]).

RaTLE COORDINATE
  The set of Ranganathan facets of an annotation, excluding [T:].
  All annotations sharing the same non-[T] facets occupy the same coordinate.
  A coordinate holds at most one RaTLE per file version.

ACQUISITION TIMESTAMP
  The date information entered the CSAR record: the session date.
  NOT the date of the described event.
  FORMAT: YYYY-MM-DD or YYYY-MM-DDTHH:MM:SS — both valid.
  Higher-granularity timestamps supersede date-only when coordinates
  are compared. A date-only timestamp is treated as YYYY-MM-DDT00:00:00
  for ordering purposes. Existing date-only timestamps are not required
  to be upgraded; both forms coexist.

THE THREE RULES

  RULE 1 — EMPTY COORDINATE: ADD
    Coordinate has never been written in the CSAR family.
    → Write annotation as per standard format. Single [T:] entry.

  RULE 2 — OCCUPIED COORDINATE: MERGE
    Coordinate exists in a prior file version.
    → Previous annotation kept unchanged in its file version.
    → New annotation written in current file version.
    → Snippet merges old + new, ordered by acquisition date, earliest first.
    → Disambiguation hint inserted at the seam.
    → Name migrates to current session provenance.
    → [T:] in facets updated to most recent acquisition date.

  RULE 3 — PREVIOUSLY MERGED COORDINATE: CASCADE
    Apply RULE 2. New entry appended. Cascade depth unbounded.

FOR BUILDERS
================================================================================

Minimum viable CSAR:
  1. File with 3 elements (calibration, output shape, nudge trigger).
  2. Two mandatory triggers (Save / Greeting).
  3. Everything else is elaboration.
When deriving: declare parent in Codex. Import modules as needed.
